### Implementation plan (plan agent, 2026-07-19)

Core design: tools NEVER write. Write tools validate and record proposals in a per-request staged dict; after the agent loop returns, the /ask handler renders staged proposals as inline-confirm messages — OWNER DECISION 2026-07-19: as SEPARATE messages after the answer (not keyboard on the answer). Confirm taps are the only write path. Cancel reuses existing rr_pause_/rr_delc_/rr_back callbacks with ZERO new callback code; only add-confirm (vrc_yes/vrc_no) is new, mirroring vtx_.

Interface needed FROM dv-ee06: (1) session runner accepting tool factories; (2) AgentToolContext: repos, user_id, currency, language, mutable staged dict; (3) runner returns answer text, handler owns sending + staged rendering; (4) system-prompt hook.

Steps:
1. domain/recurring.py: pure match_rules(rules, query) — case-insensitive token/substring, exact-match short-circuit.
2. New src/ai_tools/recurring_tools.py: build_recurring_tools(ctx) -> list_recurring (readOnlyHint, format_rules_list, NO rule IDs in output), add_recurring (item/amount required, day default 1, validate_rule_input, stage single last-wins slot), cancel_recurring (rule_ref name words; 1 match -> stage id+name; 0/many -> list candidates, stage nothing).
3. src/handlers/recurring.py: send_staged_recurring_actions (add -> AI_RECURRING_CONFIRM_ADD + vrc_yes/vrc_no keyboard, sets ai_rec_text; cancel -> rr_pause_{id}/rr_delc_{id}/rr_back) + handle_ai_recurring_confirmation (^vrc_): yes -> _inject_text("/recurring add <name> <amount> <day>") reusing full validation/copy.
4. src/core.py: register ^vrc_ BEFORE spendings_handler (~line 1033, same trap as vtx_/^rr); answer_ask_question passes build_recurring_tools + renders staged after answer.
5. texts/texts_ru: AI_RECURRING_CONFIRM_ADD / AI_RECURRING_STOP_CONFIRM / ADD_ACCEPTED / ADD_CANCELLED; existing buttons reused.
6. System prompt: writes stage-only, tell user to tap; day default 1 stated; buttons decide pause-vs-delete; ambiguous cancel -> list_recurring + ask.

Defaults accepted: apply via /recurring add injection; single last-wins staged add slot; no rule IDs / name-words-only cancel (T-027 owner decision carries over).

Tests: match_rules matrix; tools unit (staging without repo writes — spy asserts; is_error paths; day default; last-wins; no IDs); rendering/confirm keyboards + injected text; manual: DB-unchanged-until-tap, RU copy, paywall, day-31 clamp note, 0/many-match asks, voice path post dv-94bd.

Risks: multiple staged adds last-wins drop (queue = follow-up); stale staged confirm on restart (accepted, T-009); rr_delc_ one-tap delete intentional (message IS the confirm); do NOT parallel-worktree against dv-f0d5 on core.py answer_ask_question — agree AgentToolContext shape first.
