## Description

Migrated from `docs/tasks/T-037-income-in-edit-recent-entries-menu-list.md`.

Owner request 2026-07-12 (T-035 wrap-up): the menu Edit-recent-entries flow (handlers/transactions.py show_recent_entries) filters to spendings only, so income can only be deleted via /delete_income, not edited at all. Wanted: include income records in the edit list (visually marked, e.g. 💵 prefix) so they can be edited and deleted from the menu like spendings. Design constraint: the per-transaction edit keyboard offers edit category/subcategory backed by the SPENDING category dictionary — wrong for income (free-form category, no subcategory). For income rows show a reduced keyboard: edit date / edit category (free text) / edit amount / delete. Reuse repos.transactions.update/delete (type-agnostic by id) and the type-mismatch guards from T-035. Owner explicitly parked the alternative of merging /delete + /delete_income into one command: type-split commands stay, but menu editing should be unified.

## Acceptance Criteria

- [x] Edit-recent-entries list shows income records interleaved with spendings, income visually marked (💵), sorted by date desc
- [x] Income rows open a reduced edit keyboard: edit date / edit category (free text) / edit amount / delete — no subcategory
- [x] Spending rows keep today's full keyboard and behavior unchanged
- [x] Stale/forged spending-only callbacks on an income row are refused gracefully (T-035-style guard), and vice versa
- [x] Tests cover row formatting, keyboard variants, and the tx_<id> callback parsing with a mixed list

## Notes

### Implementation plan (proposed 2026-07-21)

Design: the whole edit flow lives in `src/handlers/transactions.py` and is wired as states of the spendings ConversationHandler in `src/core.py` (~lines 1005–1028: `TRANSACTION_LIST`, `TRANSACTION_EDIT`, `EDIT_DATE`, `EDIT_CATEGORY`, `EDIT_SUBCATEGORY`, `EDIT_AMOUNT`, `CONFIRM_DELETE` from `src/states.py`). `show_recent_entries` filters via `filter_by_type(session.transactions, 'spending')` (line 56) — that single line is why income is invisible. Everything downstream is already nearly type-agnostic: `handle_transaction_selection` loads by id via `repos.transactions.get_by_id` (which returns `transaction_type`), and `repos.transactions.update`/`delete` in `infrastructure/repositories/transaction_repository.py` work by id regardless of type (`transaction_type` is even in `update`'s `allowed_fields`). The type-specific parts are: (a) the per-row keyboard `create_transaction_edit_keyboard` (`src/keyboards.py:406`) offering edit category (spending dictionary keyboard, `EDIT_CATEGORY` state handles only `txcat_`/`txpage_` callbacks — no free-text path) and edit subcategory; (b) `TRANSACTION_DETAILS`/`CONFIRM_DELETE_TRANSACTION` texts that render a Subcategory line (income rows store `subcategory=""` — see `save_income_text` in `src/handlers/records.py:223`). Income categories are free-form lowercase strings (`process_income_input` defaults to `"salary"`), so income category edit must be a text prompt, not the `create_tx_categories_keyboard`. The T-035 guard pattern (`src/core.py:506` in `delete_records`: check `tx.transaction_type` and refuse mismatches) translates here to guarding `handle_edit_option` so a stale/forged spending-only callback (`edit_subcategory`, keyboard `edit_category`) can never act on an income row. One trap found: `create_numbered_transaction_keyboard` (`src/keyboards.py:568`) parses the transaction id out of the display string (`parts[0].split(': ')[0]` → callback `tx_<id>`), so the 💵 marker must go only into the message text lines, never into the `to_display_string()` values passed to the keyboard, or `int(callback_data.replace("tx_", ""))` crashes. T-044's edit-in-place navigation needs no changes: `show_recent_entries` already edits the anchor on callback, and `menu_edit_transactions` in `src/handlers/menu.py:240` just calls it. `/delete` vs `/delete_income` stay untouched (owner decision).

1. `src/handlers/transactions.py` `show_recent_entries`: replace the `filter_by_type(...)` + sort of lines 56–58 with sorting `session.transactions` (both types) by timestamp desc, keep `records_to_fetch = 30`. Extract a small pure module-level helper `format_entry_line(i, tx) -> str` that renders spending rows exactly as today and income rows as `"{i}. 💵 {date_str} - {category}: {amount} {currency}"` (no empty-subcategory colon dangling). Keep `total_amount = get_total(filter_by_type(transactions, 'spending'))` so the header sum keeps its current meaning (see open question 1). Keyboard input stays the raw `to_display_string()` list — no marker there (id-parsing trap above).
2. `src/handlers/transactions.py` `handle_transaction_selection`: add `'transaction_type'` to the `transaction` dict in both branches — `repo_transaction.transaction_type` (line 186–193) and `tx.transaction_type` for the cached-fallback (line 173–181; the domain `Transaction` in `domain/models/user_session.py` already carries it). Choose the details template by type: existing `TRANSACTION_DETAILS` for spending, new `TRANSACTION_DETAILS_INCOME` (💵 marker, no Subcategory line) for income.
3. `src/keyboards.py` `create_transaction_edit_keyboard(transaction, texts)`: branch on `transaction.get('transaction_type') == 'income'` → reduced keyboard: row [`EDIT_DATE_BUTTON` `edit_date`, `EDIT_AMOUNT_BUTTON` `edit_amount`], row [`EDIT_CATEGORY_BUTTON` `edit_income_category`], row [`DELETE_TRANSACTION_BUTTON` `delete_transaction`], row [`BACK_BUTTON` `back_to_transactions`]. Spending keyboard unchanged. Distinct callback `edit_income_category` keeps the two category paths unambiguous at the dispatch level.
4. `src/states.py`: add `EDIT_INCOME_CATEGORY` to the state tuple (`range(35)` → `range(36)`). `src/handlers/transactions.py` `handle_edit_option`: new branch for `edit_income_category` → prompt `texts.ENTER_NEW_INCOME_CATEGORY` (free text), return `EDIT_INCOME_CATEGORY`. Add the T-035-style mismatch guard at the top of the spending-only branches: if `current_transaction['transaction_type'] == 'income'` and callback is `edit_category`/`edit_subcategory` (or `== 'spending'` and callback is `edit_income_category`), `query.answer()` a short notice and re-render the details view (return `TRANSACTION_EDIT`) instead of proceeding — covers stale keyboards from before selection or forged callbacks.
5. `src/handlers/transactions.py`: new `handle_edit_income_category(update, context)` mirroring `handle_edit_amount`'s shape: read `update.message.text.strip().lower()` (same normalization as `save_income_text`), reject empty input with reprompt, `await repos.transactions.update(tx_id, user_id, category_name=new_category)`, reply `CATEGORY_UPDATED_SUCCESS`, then the standard return: `return_to_detailed` → `show_filtered_transactions`, else reset `tx_page` and `show_recent_entries`. No `load_user_session` needed (no category dictionary involved).
6. `src/core.py`: wire the new state into the spendings ConversationHandler states dict next to `EDIT_CATEGORY` (~line 1016): `EDIT_INCOME_CATEGORY: [MessageHandler(filters.TEXT & ~filters.COMMAND, handle_edit_income_category)]`; extend the `TRANSACTION_EDIT` pattern (line 1010) with `edit_income_category` (explicit alternation `^(edit_income_category|edit_date|...)`). Import `handle_edit_income_category` in the existing import block at line 147.
7. Delete confirmation: in `handle_edit_option`'s `delete_transaction` branch pick `CONFIRM_DELETE_INCOME` (new text, no Subcategory line, 💵 marker) when `transaction_type == 'income'`; `handle_delete_tx_confirmation` itself is already type-agnostic (`repos.transactions.delete` by id) — no change.
8. Texts: add `TRANSACTION_DETAILS_INCOME`, `CONFIRM_DELETE_INCOME`, `ENTER_NEW_INCOME_CATEGORY`, and the mismatch notice (reuse wording style of `DELETE_TYPE_MISMATCH`) to **both** `src/texts.py` and `src/texts_ru.py` (repo convention).
9. Tests (`tests/src/test_transaction_edit.py`, plain pytest in the T-044 `test_menu_nav.py` style using `tests/conftest.py` `make_tx`): `format_entry_line` renders 💵 + no subcategory for income and today's exact format for spending; `create_transaction_edit_keyboard` income variant has no `edit_subcategory`/`edit_category` callbacks and exactly `edit_date/edit_amount/edit_income_category/delete_transaction/back_to_transactions`; spending variant unchanged; `create_numbered_transaction_keyboard` still produces `tx_<int>` callbacks from unprefixed display strings of a mixed list. Guard test for `handle_edit_option` mismatch branch with a `SimpleNamespace`/FakeQuery like `test_menu_nav.py`.

Touched: modified `src/handlers/transactions.py`, `src/keyboards.py`, `src/states.py`, `src/core.py`, `src/texts.py`, `src/texts_ru.py`; new `tests/src/test_transaction_edit.py`. (No DB migration — `transaction_type` column and type-agnostic update/delete already exist; no changes to `/delete`/`/delete_income`, `domain/`, or `infrastructure/`.)

Open questions (recommended defaults):
1. Header total when the list mixes types → keep it as the spendings sum only (unchanged semantics; income rows are visible but not netted into the figure).
2. Normalize free-text income category to lowercase → yes (matches `save_income_text`, keeps `/show_income` aggregation buckets consistent).
3. Merged list still capped at 30 records / 2 pages → yes (income volume is low; heavy history browsing belongs to the detailed-transactions flow).

Risks: the `EDIT_INCOME_CATEGORY` state renumbers the state-int tuple in `src/states.py` — safe because states are in-memory only (no ConversationHandler persistence configured), but any in-flight conversations die on deploy restart as they already do. The detailed-transactions view (`src/detailed_transactions.py`) stays spending-filtered, yet its selection path funnels into the now type-aware `handle_transaction_selection` — income remains unreachable from there, which is intended scope. The 💵 marker must never enter the strings given to `create_numbered_transaction_keyboard` (id-parsing crash described above) — the tests in step 9 pin this. `CONFIRM_DELETE` state's `pattern="^(confirm|cancel)"` and the delete handler are shared and untouched, so a regression there would hit spendings too — covered by the review-stage manual checklist.
