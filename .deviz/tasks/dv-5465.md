## Description

Migrated from `docs/tasks/T-040-income-date-parser-accepts-garbage-99-99.md`.

Found during T-006 (bugs documented in tests/src/test_save_transaction_parsing.py with comments, current behavior pinned): (1) _parse_income_date falls back to dateutil for '99.99' and reads 99 as year 1999 — income like '99.99 1000' silently backdated 27 years; (2) '29.02' in a non-leap year fails strict dd.mm parse, dateutil reinterprets as day 29 of the CURRENT month, then backdating heuristic shifts a year back. Fix: reject the dateutil fallback when the token matched dd.mm shape but failed strict validation; update the pinned tests to the corrected behavior.

## Acceptance Criteria

- [ ] Date-shaped garbage ('99.99', '29.02' in a non-leap year, 2-digit years) is rejected with an explicit error — nothing saved, no silent backdating
- [ ] '29.02' still parses in a leap year; dd.mm and dd.mm.yyyy keep working with existing backdating rules
- [ ] Income error replies come from save_income_text (no contradictory double messages from callers)
- [ ] Pinned-bug tests in tests/src/test_save_transaction_parsing.py updated to the corrected behavior

## Notes

### Implementation plan (proposed 2026-07-21)

Design: `_parse_income_date()` (`src/save_transaction.py:109`) tries strict `%Y-%d.%m` first, then a dateutil fallback whose only legitimate job is explicit-year dates (`15.03.2024`, pinned in tests). But dateutil accepts a much larger garbage class than the two pinned bugs: `'99.99'` → 1999-07-10, `'29.02'` (non-leap) → day 29 of the current month then backdated a year, and also unpinned cases probed during planning — `'2024'` → July 2024 (income `/income 2024 1000` silently backdated two years), `'12'` → the 12th of this month, `'1.1.1'` → 2001-01-01, `'05/07'` → parsed despite the advertised format being dd.mm only (`INCOME_HELP` in `src/texts.py:43` documents exactly "date as dd.mm"). The fallback also needs an ugly double-parse `year_was_assumed` hack. So rather than bolt a shape-check veto onto dateutil, the plan replaces the fallback with a strict whitelist: `dd.mm` (existing strptime + `resolve_backdated_year`) and `dd.mm.yyyy` (`strptime("%d.%m.%Y")`, which conveniently rejects 2-digit years); a token matching the date *shape* (`^\d{1,2}\.\d{1,2}(\.\d{1,4})?$`) that fails both parses raises `ValueError` — distinct from `None` (not a date at all, e.g. `salary`). UX on rejection, decided: explicit error, nothing saved. Today a `None` return makes `process_income_input()` (line 92/100) treat the token as the *category*, so a silent fallback would save income under category "29.02" dated today — polluting the category set and dropping unambiguous date intent, the same silent-misinterpretation class this task exists to kill (cf. `domain/validation.py`'s own stance: "silently shifting it a year back is data corruption"). The single income write path is `save_income_text()` (`src/handlers/records.py:200`); its five callers (records.py x2 flows + start_income, voice.py x3) all reply the generic `TRANSACTION_ERROR_TEXT` ("You need to enter an amount…") on `False`, which would contradict a date-specific error — so error messaging is consolidated into `save_income_text` and the callers keep only state flow. The spending parser `_parse_date_to_utc()` is strict already; its silent now-fallback for `'99.99'` prefixes is pinned separately and stays out of scope. The save-path clamp (`infrastructure/repositories/transaction_repository.py:82`) continues to guard explicit future years.

1. `src/save_transaction.py` — rewrite `_parse_income_date()`: module-level `_DATE_SHAPE = re.compile(r"^\d{1,2}\.\d{1,2}(\.\d{1,4})?$")`; keep the existing strict `%Y-{text}` / `%Y-%d.%m` parse + `resolve_backdated_year` for dd.mm; add strict `datetime.strptime(text, "%d.%m.%Y")` for explicit-year (no backdating heuristic, matching `test_explicit_future_year_left_alone`); if both fail and `_DATE_SHAPE` matches → `raise ValueError(f"invalid date: {text!r}")`; otherwise return `None`. Delete the dateutil import and the `year_was_assumed` double-parse. Update the docstring: returns None for non-dates, raises ValueError for date-shaped invalid tokens.
2. `src/save_transaction.py` — `process_income_input()`: no logic change (the ValueError propagates naturally from both the len==2 and len==3 branches); update its docstring to state the raise so the contract is explicit.
3. `src/handlers/records.py` — `save_income_text()`: wrap the `process_income_input()` call in `try/except ValueError` → reply new `texts.INCOME_INVALID_DATE.format(token=parts[0])` → `return False`. Consolidate the amount-failure reply too: on the existing `float(parts[-1])` failure, reply `texts.TRANSACTION_ERROR_TEXT` before returning False (currently the callers do it). Update the docstring contract: "replies with the specific error itself; returns False when nothing was saved — callers control only conversation state."
4. Remove the now-duplicate `TRANSACTION_ERROR_TEXT` replies at the five call sites: `src/handlers/records.py` `start_income` (~line 257; on False it now just shows `INCOME_HELP`), `process_income` (~274, stays in `PROCESS_INCOME`), `process_income_menu` (~294); `src/handlers/voice.py` (~321, ~484, ~528).
5. `src/texts.py` + `src/texts_ru.py` — add `INCOME_INVALID_DATE`, e.g. EN: `"⚠️ '{token}' looks like a date but isn't a valid one (use dd.mm, e.g. 05.07). Nothing was saved — please try again."` (RU equivalent).
6. Tests — `tests/src/test_save_transaction_parsing.py`: replace the two pinned-bug tests: `test_number_like_garbage_parsed_by_dateutil` → `test_ddmm_shaped_garbage_rejected` (`pytest.raises(ValueError): _parse_income_date("99.99")`), `test_feb_29_in_non_leap_year_misparsed...` → raises ValueError under the frozen 2026 (non-leap) clock; add a `@freeze_time("2024-07-10")` case asserting `"29.02"` *is* valid in a leap year (→ 2024-02-29); add `"15.03.24"` (2-digit year) → ValueError; pin the narrowed whitelist: `"2024"`, `"12"`, `"1.1.1"` → `None`; add a `process_income_input` propagation test (`["29.02", "1000"]` raises). Existing dd.mm / dd.mm.yyyy / backdating tests must pass unchanged.

Touched: modified `src/save_transaction.py`, `src/handlers/records.py`, `src/handlers/voice.py`, `src/texts.py`, `src/texts_ru.py`, `tests/src/test_save_transaction_parsing.py`. (No DB, config, or requirements changes; `python-dateutil` stays a dependency — it is used elsewhere.)

Open questions (recommended defaults):
1. Narrow the parser to exactly dd.mm and dd.mm.yyyy, dropping dateutil's unadvertised freebies (`'12'` → 12th of this month, `'2024'`, ISO strings, `'05/07'`) → yes (INCOME_HELP advertises dd.mm only; every dropped form is also a plausible amount/category, which is precisely the ambiguity that produced this bug).
2. Consolidate all income error replies into `save_income_text` (5 call sites stop sending `TRANSACTION_ERROR_TEXT`) → yes (avoids a contradictory double message "invalid date" + "you need to enter an amount"; callers keep state flow only).

Risks: the whitelist is a behavior narrowing — any user habitually typing an unadvertised dateutil-only format (e.g. `12 1000` for "the 12th") now gets that token saved as a category, visible in the `INCOME_SAVED` confirmation; acceptable and easily reversed by whitelisting more formats later. `process_income_input` gains a raising path — its only runtime caller is `save_income_text` (the `src/core.py:68` import is a re-export with no call site), but any future caller must handle ValueError, which the docstring will state. In the voice fix flow (`handle_voice_fix_confirmation`, `src/handlers/voice.py`), the old row is deleted *before* the replacement save, so a date-invalid fix payload now aborts after deletion — this delete-then-fail window is pre-existing (amount failures hit it today) and the user now gets a clearer error; worth a follow-up task, not this one. `tests/src/test_voice_routing.py` and `tests/src/test_menu_nav.py` may exercise the touched caller lines and need their expected-reply assertions adjusted if they pin the removed `TRANSACTION_ERROR_TEXT` replies.
